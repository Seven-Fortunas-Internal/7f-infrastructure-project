## 5. Agent Rules

### 5.1 Verification Results Format (CRITICAL)

When updating `feature_list.json` after implementing a feature, the
`verification_results` values **must** be exactly the lowercase string `"pass"` or
`"fail"`. No prose. No `"PASS - description..."`. No `null`.

```bash
# ✓ CORRECT
--arg func "pass" --arg tech "pass" --arg integ "pass"

# ✗ WRONG — causes CI failures
--arg func "PASS - sprint workflows operational"
--arg func ""
```

If a test cannot be run: set the value to `"skipped"` and set feature
`status` to `"blocked"` with a `blocked_reason`.

### 5.2 What Agents Must NEVER Create at Root

The following file types must **never** be created at the project root
(`/home/ladmin/dev/GDF/7F_github/`):

| Type | Examples | Rule |
|------|---------|------|
| Log files | `*.log`, `*-run.log`, `issues.log` | Use `/tmp/` for temporary logs |
| Backup files | `*.backup.*`, `*.bak`, `*~` | Never create backups; git is your backup |
| One-time scripts | `verify_*.sh`, `extract_*.py`, `parse_*.py` | Use `scripts/` or `tests/` |
| Duplicate data | `app_spec.txt` (root), `planning-artifacts/` (root) | Use canonical locations (Section 3) |
| Session reports | `SESSION-*.md`, `*_report.md` | Archive to `_bmad-output/archive/` |
| Old session logs | `init-session*.log`, `session2*.log` | These are debris — don't create them |

**If you generate a file that doesn't fit the directory structure in Section 4,
stop and decide where it actually belongs before writing it.**

### 5.3 What the Agent Reads — Path Reference

```bash
# Feature specification (READ ONLY — never modify)
_bmad-output/app_spec.txt

# Progress tracking (read/write)
feature_list.json          # jq queries only — never cat the whole file
claude-progress.txt        # sed updates only

# Quality gate scripts (run before marking any feature pass)
scripts/validate-and-fix-workflow.sh   # For .github/workflows/*.yml files
```

### 5.4 Quality Gate Before Marking "pass" (MANDATORY)

For every `.github/workflows/*.yml` file the agent generates:

```bash
VALIDATOR="scripts/validate-and-fix-workflow.sh"
if [[ -x "$VALIDATOR" ]]; then
  bash "$VALIDATOR" ".github/workflows/YOURWORKFLOW.yml"
  # Only mark feature "pass" if this exits 0
fi
```

NFR-5.6 is a **constraint on every workflow you generate**, not a one-time
deliverable. Run the validator every time.

### 5.5 Workflow Naming Rules

Before writing any `.github/workflows/*.yml` file:

1. Check for name collision:
   ```bash
   grep -rh "^name:" .github/workflows/*.yml 2>/dev/null | sort
   ```
2. Your intended `name:` field must not already exist.
3. Use the format: `"Action Verb + Noun + (Optional Phase Label)"` e.g.
   `"Collect SOC 2 Evidence"`, `"Validate Workflow Compliance (FR-10.1)"`.

### 5.6 Workflow Sentinel — Write It Last

The Workflow Sentinel (`workflow-sentinel.yml`) **must be the last workflow
created** in any session. Before writing it, generate the watch list from
actual deployed workflow names:

```bash
NAMES=$(grep -rh "^name:" .github/workflows/*.yml 2>/dev/null \
  | sed 's/^name: //' | sort -u)
# Use $NAMES as the workflows: list in workflow-sentinel.yml
```

Never hardcode guessed names. Always read actual names from disk.

### 5.8 Tracking Files Are Gitignored — Never Sync From Remote

`feature_list.json`, `claude-progress.txt`, and `autonomous_build_log.md` are
**gitignored**. They do NOT exist on `origin/main` or any remote branch.

**⛔ NEVER run:**
```bash
git show origin/main:feature_list.json   # FORBIDDEN — file is not on remote
```

The **local** `feature_list.json` is the ONLY source of truth for feature status.
If local shows 0 pending features → the session is complete. Do not look at remote.

**Fresh start before every autonomous run:** delete all three files and let the
initializer regenerate them with all features set to `pending`.

### 5.9 Active GitHub Account Must Be `jorge-at-sf`

Before any `gh` command, verify:
```bash
gh api user --jq '.login'  # Must return: jorge-at-sf
```

If wrong account: `gh auth switch --user jorge-at-sf`

This is enforced by `run-autonomous.sh` pre-flight AND by the coding agent's STEP 1
orientation. Defense in depth — both checks must pass.

### 5.7 Artifact Filename Sanitization

Never use ISO 8601 timestamps with colons in filenames or artifact paths:

```bash
# ✓ CORRECT
artifact_name: "soc2-evidence-${{ github.run_number }}"
artifact_name: "report-$(date +%Y-%m-%d)"

# ✗ WRONG — colons are invalid in artifact paths
artifact_name: "evidence-$(date -u +%Y-%m-%dT%H:%M:%SZ)"
```

## 6. PR Review — When to Flag for Human Review

PRs opened by `jorge-at-sf` are auto-approved by `bot585`. The bot satisfies the
process gate only — it does not read code. Agents must flag the following change
types to Jorge for manual review **before merging**:

| Flag if PR touches | Reason |
|--------------------|--------|
| `.github/workflows/` | Workflow change can subvert the CI gate itself |
| `auto-approve-pr.yml` | Modifies the approval mechanism |
| Branch protection settings | Changes what gates a merge |
| Any secret, auth, or access control file | High blast radius |
| `compliance/` or `scripts/security*` | SOC 2 evidence chain |
| Major dependency version bumps | Supply chain risk |

For all other change types, bot approval + green CI is sufficient.

Full reference: `docs/devops/pr-review-practices.md`

- Always include `Co-Authored-By` line

## 9. Bounded Retry Strategy

If a feature fails:

1. **Attempt 1:** Full implementation per spec
2. **Attempt 2:** Simplified — core functionality only
3. **Attempt 3:** Minimal viable — bare essentials
4. **Attempt 4+:** Mark `"blocked"`, document reason, move on

Do not get stuck. Move to the next feature.

| `gh api` returns HTTP 401 | `gh auth login && gh auth status` |
